# Правила работы с репозиторием

## Перед каждой задачей

1. Всегда прочитать `AGENTS.md`.
2. Если задача касается игровых механик — прочитать `docs/GAME_DESIGN.md`.
3. Если задача касается архитектуры — прочитать `docs/ARCHITECTURE.md`.
4. Не перечитывать остальные документы без необходимости.

## Общие правила

1. Не переписывать весь проект ради небольшой задачи.
2. Не удалять работающие функции без прямого указания.
3. Вносить минимальные и понятные изменения.
4. Не использовать тяжёлые фреймворки, сборщики, npm и внешние зависимости без явного разрешения.
5. Проект должен запускаться как обычный статический сайт на GitHub Pages.
6. Использовать HTML, CSS и чистый JavaScript.
7. Не хранить секреты, токены, пароли и персональные данные в репозитории.
8. Не менять файлы вне текущего репозитория.
9. Перед завершением проверять синтаксис изменённых JavaScript-файлов и отсутствие очевидных ошибок.
10. Сохранять поддержку мобильных экранов.
11. Не создавать огромные файлы: разделять код по назначению, когда это действительно нужно.
12. Игровое состояние хранить отдельно от отображения интерфейса.
13. Не смешивать в одном месте игровую логику, работу с DOM и данные.
14. При изменении архитектуры обновлять `docs/ARCHITECTURE.md`.
15. При добавлении или изменении игровой механики обновлять `docs/GAME_DESIGN.md`.
16. После каждой завершённой задачи добавлять краткую запись в `docs/CHANGELOG.md`.
17. В итоговом ответе всегда указывать:
    - что изменено;
    - какие файлы затронуты;
    - как проверить результат;
    - какие ограничения или нерешённые вопросы остались.
18. Не делать commit и push самостоятельно: этим занимается Telegram-мост.
19. Если запрос неоднозначен и есть риск существенно изменить проект не так, сначала описать безопасное предположение или задать уточняющий вопрос, а не придумывать крупные решения.
