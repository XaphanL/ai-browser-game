# Архитектура проекта

## Текущее состояние

Проект является статической браузерной игрой без сборки и внешних зависимостей. `index.html` содержит каркас интерфейса, стили вынесены в `css/styles.css`, а игра реализована ES-модулями на чистом JavaScript и Canvas 2D.

Текущая структура:

```text
.
├── index.html              # каркас игрового экрана и мобильных контролов
├── css/styles.css          # интерфейс, адаптивность и экранные контролы
├── js/
│   ├── main.js             # запуск цикла и обработка игровых событий
│   ├── config.js           # параметры баланса игрока, комнат и босса
│   ├── state.js            # создание игрового состояния
│   ├── game.js             # игровая симуляция без работы с DOM
│   ├── input.js            # клавиатура, мышь и сенсорные кнопки
│   ├── renderer.js         # отрисовка состояния на Canvas
│   └── ui.js               # обновление HTML-интерфейса
├── README.md               # описание и инструкции по запуску
├── AGENTS.md               # правила работы с репозиторием
├── css/                    # будущие таблицы стилей
├── assets/
│   ├── images/             # будущие изображения
│   └── audio/              # будущие аудиоматериалы
├── docs/                   # игровая и проектная документация
└── tests/                  # будущие проверки
```

В корне также сохранены существующие служебные и проверочные файлы `TEST.md`, `github_push_test.txt` и `hello.txt`. Они не участвуют в работе страницы.

Пустые каталоги отмечены файлами `.gitkeep`, чтобы Git сохранял структуру. Эти файлы не являются исходным кодом.

## Разделение ответственности

`state.js` создаёт обычный объект состояния комнаты и не отвечает за отображение. Тип комнаты (`arena` или `boss`), состояние восьми граней брони и параметры противников хранятся в игровом состоянии. `game.js` изменяет это состояние по входным данным и возвращает события переходов. `renderer.js` только рисует состояние на Canvas, включая броню и индикатор здоровья босса, а `ui.js` обновляет DOM. `main.js` связывает компоненты и управляет переходом между комнатами. Такая граница позволяет развивать механику отдельно от интерфейса.
